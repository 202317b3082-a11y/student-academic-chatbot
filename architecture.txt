# Project Architecture Overview

This document provides a complete textual overview of how the **Student Academic Chatbot** system works, describing each component, their interactions, data flow, and deployment considerations.

## 2. Main Components

| Component | Path / File | Responsibility |
|-----------|------------|----------------|
| **Static Front‑end** | `public/` (HTML, CSS, JS) | UI for login, signup, chat, admin dashboard. Communicates with backend via AJAX (`fetch`). |
| **Flask Application** | `flask_app.py` | Sets up routes, JWT handling, request validation, and ties all services together. |
| **Authentication Service** | `flask_app.py` (login / signup endpoints) & `db.py` | Stores user credentials in `users.db`, hashes passwords, issues JWTs containing role claims. |
| **Business Logic** | `logic_response.py` | Receives a query, creates an embedding, searches the FAISS index, formats the answer, and calculates confidence. |
| **Suggestion Engine** | `suggestion.py` | Uses a second FAISS index (`faiss_feedback_index`) built from historical queries/feedback to suggest similar past questions. |
| **Embedding Generation** | `create_embedding.py` & `feedback_embedding.py` | Convert raw text (`college_policy.txt` and `feedback.csv`) into dense vectors using **sentence‑transformers/all‑MiniLM‑L6‑v2** and store them in FAISS indexes. |
| **Feedback Collector** | `flask_app.py` (`/feedback` endpoint) | Persists user rating together with the original question/answer in `feedback.csv`. |
| **Evaluation Suite** | `evaluation.py` | Runs offline metrics (confidence, precision/recall, satisfaction) for quality monitoring. |
| **Database Layer** | `db.py` | SQLite wrapper for `users.db`. Simple CRUD helpers used by Flask routes. |
| **Configuration** | `flask_app.py` & optional `.env` | Secret key, JWT expiry, debug mode, optional environment variables for production. |
| **Cache** | `hf_cache/` | Stores downloaded transformer model files to avoid re‑downloading. |

## 3. Data Flow for a Typical User Query
1. **User Interaction** – The student opens `ChatUI.html`, types a question and clicks *Send*.
2. **Front‑end Request** – JavaScript sends a `POST /chat` request with the JWT in the `Authorization` header and the question in the JSON body.
3. **Authentication Check** – `flask_app.py` validates the JWT, extracts the user role, and forwards the request to the business logic.
4. **Embedding Creation** – `logic_response.py` calls the Sentence‑Transformer model to embed the incoming question.
5. **Semantic Search** – The embedding is queried against the primary FAISS index (`faiss_index/`). The top‑k most similar passages are retrieved.
6. **Answer Generation** – The retrieved passages are stitched together and (optionally) passed to an LLM (if integrated) to produce a final answer and a confidence score.
7. **Suggestion Retrieval** – Simultaneously, the same question embedding is sent to the feedback FAISS index (`faiss_feedback_index/`) to obtain similar historic queries. Those are returned as *suggestions*.
8. **Response Packaging** – `flask_app.py` returns a JSON payload containing:
   - `answer`
   - `confidence`
   - `suggestions` (array of similar past questions)
   - `source_passages` (optional for debugging)
9. **Front‑end Rendering** – The UI displays the answer, confidence bar, and clickable suggestion snippets.
10. **Feedback Loop** – The user may click *👍* or *👎*. The front‑end sends a `POST /feedback` request with the original question, answer, and rating. The backend appends the record to `feedback.csv`.
11. **Embedding Update** – Periodically (or manually) the maintainer runs `feedback_embedding.py` to rebuild the suggestion FAISS index, incorporating the new feedback data.

## 4. Admin Workflow
1. Admin logs in via `admin.html`. After successful JWT validation, Flask redirects to the admin dashboard.
2. Dashboard UI calls protected endpoints:
   - `GET /users` – list all users.
   - `POST /user/<email>/role` – change role.
   - File upload endpoint (custom) – replace `college_policy.txt`.
3. After updating the knowledge‑base, the admin runs `python create_embedding.py` to regenerate the primary FAISS index.
4. Admin can also trigger `python evaluation.py` to view metric reports.

## 5. Deployment & Production Considerations
- **WSGI Server** – Use **Gunicorn** (e.g., `gunicorn -w 4 -b 0.0.0.0:5000 flask_app:app`).
- **Environment Variables** – Store `FLASK_SECRET_KEY`, `JWT_EXPIRY_HOURS`, and any DB paths in a `.env` file; load with `python-dotenv`.
- **HTTPS** – Terminate TLS at a reverse proxy (NGINX/Apache) to protect JWTs.
- **Scaling** – The heavy part is the embedding model; consider serving it via a separate micro‑service (FastAPI) or using a GPU‑enabled instance.
- **Persistence** – `users.db` and `feedback.csv` can be migrated to a managed RDBMS or cloud storage for larger deployments.
- **Backup** – Periodically back up `faiss_index/`, `faiss_feedback_index/`, and SQLite files.

---

## 6. Extensibility Checklist
- **Add New Knowledge Sources** – Drop a `.txt` file, run `create_embedding.py`, optionally expose an admin UI to manage multiple sources.
- **Swap LLM Backend** – Replace the answer stitching step in `logic_response.py` with a call to OpenAI, Anthropic, or a locally hosted model.
- **Introduce Role‑Based Permissions** – Extend JWT payload and Flask decorators to protect additional routes.
- **Realtime Metrics Dashboard** – Feed `evaluation.py` results into Grafana/Prometheus for live monitoring.
- **Containerisation** – Dockerfile can encapsulate the Flask app, FAISS indexes, and model cache for reproducible deployments.
